# Self-Study Material Generator

## 任务定位

为指定群体编写**完全自足**的自学材料。"自足"指：学习者无法向作者提问，材料必须就地解释一切，使认真学习者能独立达到既定能力目标。

## 何时使用本 skill

匹配场景的典型特征：

- 用户明确要"编写讲义/教程/教材/手册"等长篇结构化教学材料；
- 用户指定了学习者群体（如"面向某专业本科生""面向有 Python 基础的产品经理"）；
- 内容需要分章节、有学习路径，而非单点知识问答；
- 学习者在学习过程中得不到答疑，需要材料自洽。

不匹配的场景：写博客、回答单一问题、做演示文稿（应触发 pptx skill）、直接的代码任务。

## 进度笔记机制（贯穿全程，必做）

本类任务篇幅长、跨多轮对话，上下文常被压缩（compact）后续作。为保证压缩或换会话后能**无损接续**，须维护一份进度笔记，**每完成一次操作（每写完一章、每修订一节、每轮审校、每次格式化）后立即更新它**，不得拖到阶段末尾。

- **位置**：与交付物同目录，命名 `progress_notes.md`（如交付目录尚未确定，先放工作目录，确定后随交付物一并迁移）。该文件是工作产物、不对外交付，最终交付时不复制到 `outputs/`。
- **何时写**：阶段 0 收齐参数后创建；此后每完成一个可见操作就追加/更新；每个阶段切换时落一条小结。
- **写什么**（覆盖式维护，保持精简、最新即可，不堆历史流水）：
  1. **固定参数**：学科、受众、先修、阶段划分、能力目标、输出格式、密度/字数等量化目标；
  2. **大纲决议**：章节清单、样题编号与归属章、范围外清单、用户对大纲的修改意见；
  3. **逐章状态表**：每章 `章号 | 标题 | 状态(未写/初稿/已扩写/已审校) | 篇幅(含代码行) | 例题数 | 习题数 | 备注`；
  4. **待办队列**：尚未完成的章节与阶段（如"阶段 3 四轮审校未做""阶段 4 HTML 未构建"）；
  5. **决策与约定**：术语/符号约定、语体要求、用户反馈与纠正（含原因），以及踩过的坑与修法；
  6. **下一步**：明确写出"续作时应做的第一件事"。
- **怎么用**：每次续作（尤其压缩后）**先读 `progress_notes.md`** 再动手，以它为权威进度来源，避免重复劳动或遗漏待办。
- **取舍**：笔记服务于接续，不是日志归档；信息过期即覆盖，保持单一可信视图。

## 执行流程

### 阶段 0：参数收集

从用户首次请求中提取下列要素。**缺失项必须向用户追问，不要猜测**：

| 必填要素 | 说明 |
|---|---|
| 学科 / 主题 | 材料涵盖的领域 |
| 目标受众 | 学习者身份、专业背景、所处阶段 |
| 学习者先修水平 | 可假定已掌握什么（避免冗余复习） |
| 阶段划分 | 是否分多个学期 / 模块；前后段定位 |
| 最终能力目标 | 学完后学习者能做什么 |
| 输出格式 | markdown / html / 两者兼有 / 其他 |

可选要素（缺失时按合理默认值处理，但应在大纲阶段告知用户）：

- 不在范围内的内容（默认：由 agent 推断并在大纲阶段列出请用户确认）
- 量化质量目标（默认：通过 80+ 难度的综合考核）
- 目标难度样题（默认：由 agent 根据学科与最终能力目标设计 2–3 道，在大纲阶段提交确认）
- 内容密度（默认见 references/content_density.md）

收集策略：**一次性追问所有缺失的必填要素**，不要挤牙膏式逐条提问。

### 阶段 1：大纲产出

产出包含以下内容的大纲：

1. 全部章节标题与每章学习目标（一两句话）；
2. 每章对应的核心概念清单；
3. 每章预估的例题与习题数量；
4. 每章对应支撑的"目标难度样题"编号；
5. agent 自行设计的 2–3 道目标难度样题（如用户未提供）；
6. agent 推断的"不在范围内"清单（如用户未明示）。

大纲产出后**必须停下**，等待用户确认或修改，方可进入下一阶段。

### 阶段 2：章节产出

按大纲顺序产出章节内容。**每次只产出 1–2 章**，产出后停下等待确认。不允许一次性输出全部章节——长上下文中后续章节质量会显著衰减。

每章必须包含的结构：

- 开篇导入：本章解决什么问题、与前后章的关系；
- 核心概念展开：使用前定义，定义前给动机；
- 例题：完整解答，展示思考过程而非只给答案；
- 关键定理 / 结论：附证明梗概或可执行的证明思路，不允许仅有陈述；
- 习题：分级（基础 / 提高 / 挑战），每题附答案、提示或参考解法；
- 本章自检清单：学完后应能完成的具体任务。

密度参考见 `references/content_density.md`。

**语体限制**：全文使用严肃、专业的书面语。不使用绘文字（emoji）、颜文字、网络流行语等非严肃表达——这是面向自主学习者的教材，语气的随意会削弱内容的可信度与专业感。此限制不仅适用于 markdown 正文，也适用于 HTML 界面文案（导航按钮、提示语等）；`templates/html_template.html` 与 `templates/build_html.py` 中的图标/按钮文字同样只用无装饰性的功能性符号（如 ☰、←、→），不用 emoji。

### 阶段 3：审校

全部章节产出后，进入审校。**分四轮独立执行**，每轮先输出问题清单等用户确认，再修订。绝不在审校阶段静默重写。

四轮顺序与检查项详见 `references/audit_checklist.md`：

1. 事实 / 正确性 —— 概念、公式、人名、年份、引文、数值；
2. 内部一致性 —— 符号、术语、记号约定、跨章引用；
3. 教学性 —— 例题覆盖、难度梯度、过渡衔接、对样题的支撑度；
4. 格式 / 排版 —— 渲染、目录、链接、代码块、图表对齐。

### 阶段 4：格式化与交付

按用户指定格式输出。技术规范与交付物目录结构详见 `references/format_specs.md`。

**标准交付包**（除非用户明确只要其中一部分，三层都要交付）：

1. **逐章 markdown**：`chapters/chNN.md`，每章一个文件（前言、附录也各自独立成文件）；
2. **逐章 HTML**：`chapters_html/chNN.html`，用 `templates/build_html.py` 对每个 md 单独构建，标题取该文件首行 `#` 标题；每章页面顶部有锁定在视口顶端、随滚动始终可见的导航横幅（标题居中，上一章/下一章分居两侧、章节名居中显示且不带章序号），底部另有一条随文排布的"上一章 / 目录 / 下一章"收尾导航条，两处都支持左右方向键翻章，使读者无需每次回到目录页即可连续阅读；另生成 `chapters_html/index.html` 目录页，链接到全部分章页面与合并版；
3. **合并版**：`front_matter + 各章 + 附录` 按序拼接为单一 md，再构建单一 HTML（利用模板的目录侧栏做全书导航）。

若需 html：使用 `templates/build_html.py` 把 markdown 转为 html。脚本与 `templates/html_template.html` 配套使用，已内置 MathJax、目录侧栏、暗色模式、可打印样式、章节间导航条。调用方式：

```bash
python3 templates/build_html.py input.md output.html --title "..." --header "..."
```

或在 Python 中：

```python
from templates.build_html import convert
convert('input.md', 'output.html', title='...', header='...')
```

构建**逐章 HTML** 时，额外传入 `--prev-href/--prev-title`、`--next-href/--next-title`、`--index-href` 即可让该页面出现"上一章 / 目录 / 下一章"导航条（不传则不渲染，合并版/目录页本身应保持不传）。完整的按阅读顺序批量构建脚本见 `references/format_specs.md`「逐章 HTML 构建」一节，照抄即可。

依赖：`pip install markdown`

**合并时的坑**：直接 `cat` 拼接时，若前一文件末尾无空行，下一文件的 `# 标题` 会紧贴前文（尤其是表格行），导致 python-markdown 不把它解析为标题。务必用"每个文件后补一个空行"的方式拼接（具体命令见 format_specs.md）。

最终交付：将交付包（合并版 md/html + `chapters/` + `chapters_html/` + 大纲等）复制到交付目录（云端环境为 `/mnt/user-data/outputs/` 并调用 `present_files`；本地环境为用户家目录下以主题命名的目录，如 `~/<topic>/`）。`progress_notes.md` 不随交付。

## 不确定性处理

对事实、人名、年份、公式系数、定理归属、引文卷期等不确定的内容：

- 必须明确标注 `【待核实】` 而非猜测；
- 参考文献只列高度确信存在的条目；
- 整个话题超出可靠知识范围时，直接说明并建议用户补充权威资料；
- 涉及当前事实（如某领域最新进展、最新版本号），调用 web_search 验证而非依赖记忆。

## 与用户的交互原则

- **每次操作后更新进度笔记**：每写完一章、每修订、每轮审校后立即更新 `progress_notes.md`（见上节"进度笔记机制"），保证压缩或换会话后可无损续作。
- **每阶段停下等待**：大纲、每 1–2 章、每轮审校之后都要停。这是质量保证机制，不要为图省事跳过。
- **缺失参数主动追问，不擅自决定**：尤其是受众与最终能力目标这两项，决定了整份材料的深度。
- **变更范围时确认**：用户中途要求增删章节、改变难度时，先回到大纲层面确认，再继续。
- **承认知识边界**：宁可标注"待核实"或建议外部参考，也不要编造。
